Chopstick

Jekyll boilerplate, Gulp workflow and SCSS framework

Overview

  • Jekyll boilerplate
    Quickly build HTML prototypes with partials, markdown, data sets and more
  • Gulp workflow
    Automation: local server, browser sync and file watch, js concatenation, css pre- and postprocessing
  • SCSS framework
    Solid architecture using 'sensible' defaults, single-depth self-documenting selectors and namespaces

Dependencies

  • Ruby (+ RVM – Ruby version manager)
  • Node.js and NPM (+ NVM – Node version manager)
  • Gulp.js and Bower.js as system installs on npm
  • There is a install.sh script available
    Warning: not extensively tested.

SCSS Framework

A framework, not a UI-kit!

  • Maintain sensible defaults
  • Enforce a global and clear coding style
  • Create a scalable system for both small and big web projects

How?

  • ITCSS architecture (scalable)
  • Low specificity (single-depth classes)
  • Self documenting (namespaces, BEM)
  • Open/closed principle (add via extension, not modification)
  • SCSS as preprocessor for variables and mixins, PostCSS(Autoprefixer) as post-processsor.

ITCSS architecture

  • Inverted triangle CSS = a logical and scalable way to organise selectors
  • Uses layers that organise selectors from low specificity to high specificity
  • Naming convention: easy to find and add selectors

Layers

  • Settings: contains global variables and config switches
  • Tools: contains default mixins and functions
  • Generic: contains ground-zero styles (Normalize.css, resets, box-sizing)
  • Objects: contains cosmetic-free design patterns
  • Components: contains designed components, chunks of UI
  • Plugins: contains third-party plugin styles
  • Trumps: contains helpers, overrides and shame.css

In use

SCSS is used to import partials.
Filenames start with the layer name. 

    
    // Example layers
    @import "settings.global";
    @import "tools.functions";
    @import "generic.normalize";
    @import "elements.page";
    @import "objects.grid";
    @import "components.navigation";
    @import "plugins.googlemaps";
    @import "trumps.widths";
    @import "trumps.shame";

Low specificity

Faster, easier to read, easier to modify and extend

  • Minimal styling of HMTL elements.
  • Absolutely no styling with ID's!
  • Only single-depth classes (no nesting, no scoping)
  • States (eg. 'is-active') are exceptions and can be more specific

Self documenting classnames

  • Namespaces
  • BEM (Block__element--modifier)

Namespaces

Prefix classnames depending their function and location in the architecture

  • Object: '.o-*'
  • Component: '.c-*'
  • Plugin (when possible): '.p-*'
  • Utility: '.u-*'
  • State: '.is-*' or '.has-*'
  • Javascript hook: '.js-*'

Open/closed principle

Add via extension, not modification!

  • Do not edit an object unless it is a bugfix
  • Do write modifiers (--modifier) on existing objects and components
  • Add new objects, new components when necessary
  • Use existing utility classes or write new ones for edge cases or quick fixes
